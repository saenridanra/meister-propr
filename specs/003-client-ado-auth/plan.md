# Implementation Plan: Per-Client ADO Identity

**Branch**: `003-client-ado-auth` | **Date**: 2026-03-09 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/003-client-ado-auth/spec.md`

## Summary

Attach optional ADO service principal credentials (tenant ID, client ID, secret) to the `ClientRecord` so each customer organisation can supply its own identity. All ADO operations (PR crawl, review fetch, comment posting, identity resolution) automatically use the client's credentials when present, falling back to the global backend identity when absent. Credentials are managed via two new admin endpoints (`PUT`/`DELETE /clients/{id}/ado-credentials`) and are never returned in API responses.

## Technical Context

**Language/Version**: C# 13 / .NET 10, TFM `net10.0`
**Primary Dependencies**: ASP.NET Core MVC, EF Core 10.0.3, Npgsql 10.0.0, Azure.Identity 1.14.2, Microsoft.TeamFoundationServer.Client 20.269.0-preview
**Storage**: PostgreSQL 17 via EF Core (three new nullable columns on `clients` table)
**Testing**: xUnit + NSubstitute; `WebApplicationFactory<Program>` for integration tests
**Target Platform**: Linux rootless container
**Project Type**: Web service (REST API)
**Performance Goals**: No new hot paths; credential lookup is a single PK read (sub-millisecond)
**Constraints**: Secret never leaves the server; no new env vars; backward-compatible migration
**Scale/Scope**: One credential record per client; number of clients expected in the low hundreds

## Complexity Tracking

| Violation | Why Needed | Simpler Alternative Rejected Because |
|---|---|---|
| `IIdentityResolver.ResolveAsync` signature change (new `clientId` param) | Identity lookup must use the client's SP when present; `clientId` is the only key available to the Application layer | Passing `ClientAdoCredentials?` directly would leak Azure.Core types into the Application interface; a separate `IClientAdoIdentityResolver` would duplicate interface hierarchy for no gain |
| Two new admin endpoints instead of extending `PATCH /clients/{id}` | `PUT` / `DELETE` aligns with REST sub-resource semantics; keeps `PATCH` focused on `IsActive` only; avoids partial-update ambiguity for the atomic credential group | Extending PATCH with nullable credential fields would require complex "null = don't change, empty string = clear" logic that is error-prone and hard to document clearly |
