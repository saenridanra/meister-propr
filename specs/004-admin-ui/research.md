# Research: Admin Management UI (004-admin-ui)

**Feature**: 004-admin-ui | **Date**: 2026-03-09

---

## Decision 1: Frontend Framework

**Decision**: Vue 3 (Composition API, `<script setup>`) with Vite 6

**Rationale**: User explicitly specified Vue + npm. Vite is the standard build tool for Vue 3
projects and provides fast HMR, built-in dev proxy (eliminating CORS for local dev), and an
optimised multi-chunk production build.

**Alternatives considered**:
- React + Vite — rejected (user chose Vue)
- Nuxt 3 — rejected (SSR overhead unnecessary; adds complexity for a local admin SPA)
- Vue 2 + Webpack — rejected (EOL; Vite is the modern standard)

---

## Decision 2: State Management

**Decision**: Vue 3 `ref`/`reactive` + composables only. No Pinia or Vuex.

**Rationale**: The SPA has minimal global state — only the admin key (sessionStorage) and the
loaded client list. A single `useSession.ts` composable covers auth state; `ClientsView.vue`
holds the local list state with `ref`. Adding Pinia for this scope would be premature complexity.

**Alternatives considered**:
- Pinia — considered; rejected because there are only 2 pieces of global state (session key and
  client list). Can be added if scope grows.
- Vuex 4 — rejected (superseded by Pinia; overly verbose for this use case)

---

## Decision 3: HTTP Client & API Type Generation

**Decision**: `openapi-typescript` (dev dependency, code generation) + `openapi-fetch` (runtime typed client).

**Rationale**: The backend already commits `openapi.json` to the repo root (generated by Swashbuckle
on every endpoint change per constitution principle I). `openapi-typescript` converts that schema into
a fully-typed TypeScript `paths` object with zero runtime overhead. `openapi-fetch` wraps native
`fetch` with those types so every API call is checked against the actual backend contract at compile
time — catching endpoint, parameter, and response shape mismatches before they reach the browser.
This replaces a hand-written `api.ts` fetch wrapper and eliminates the risk of type drift between the
frontend and backend.

`api.ts` is reduced to a thin factory: `createAdminClient(opts?)` that creates an `openapi-fetch`
client, attaches a request middleware to inject `X-Admin-Key`, and attaches a response middleware to
handle 401 (clear session → throw `UnauthorizedError`). The `overrideKey` option enables the login
flow to verify a candidate key before writing it to sessionStorage, without bypassing the typed
client.

**npm scripts**:
```
"generate:api": "openapi-typescript ../../openapi.json -o src/services/generated/openapi.ts"
```
Run after any backend endpoint change; generated file committed to source control.

**Alternatives considered**:
- Hand-written `api.ts` with native `fetch` — rejected (type drift risk; duplicates schema already in `openapi.json`)
- Axios — rejected (extra runtime weight; openapi-fetch uses native fetch)
- `@hey-api/openapi-ts` — considered; rejected (generates more files, more configuration surface; openapi-typescript + openapi-fetch is simpler and more composable)
- `orval` — rejected (pulls in TanStack Query dependency; over-engineered for this use case)

## Decision 4: CORS Strategy

**Decision**: Vite dev server proxy for local development; same-origin nginx for Docker.

**Rationale**: The Vite proxy transparently forwards `/clients`, `/reviews`, `/identities`, and
`/healthz` from `http://localhost:5173` to the backend on `http://localhost:8080`. From the
browser's perspective, all requests stay on the same origin — no CORS headers required. In
Docker, nginx serves the SPA's static files under `/admin/` on the same `https://localhost:5443`
host as the backend API — again same origin, no CORS needed.

The existing `CORS_ORIGINS` env-var mechanism remains available as a fallback for deployments
that serve the SPA from a different host (e.g., a CDN), but no backend code changes are required.

**Vite proxy targets** (configured in `vite.config.ts`):

| Path prefix  | Proxied to                 |
|--------------|----------------------------|
| `/clients`   | `http://localhost:8080`    |
| `/reviews`   | `http://localhost:8080`    |
| `/identities`| `http://localhost:8080`    |
| `/healthz`   | `http://localhost:8080`    |

**Alternatives considered**:
- Add `http://localhost:5173` to backend fixed CORS origins — rejected (pollutes production
  config; the Vite proxy approach is cleaner and already industry standard)
- BFF (Backend-for-Frontend) proxy — rejected (over-engineering for a single-tenant admin UI)

## Decision 5: Admin Key Session Storage

**Decision**: `sessionStorage` keyed at `"meisterpropr_admin_key"`.

**Rationale**: `sessionStorage` is cleared automatically when the browser tab is closed,
matching the "short-lived credential" requirement in the spec. `localStorage` would persist
across sessions (undesirable for an admin key). A cookie-based approach would require
additional backend middleware; `sessionStorage` keeps the solution frontend-only and simpler.

The SPA verifies the key at login time by calling `GET /clients` with the entered key in
`X-Admin-Key`. A `200 OK` confirms validity; `401` shows an error and does not store the key.
On any subsequent `401` response the key is cleared and the user is redirected to the login view.

**Alternatives considered**:
- `localStorage` — rejected (persists across restarts; longer exposure window)
- Cookie with `HttpOnly` + `SameSite=Strict` — rejected (requires backend session management;
  over-engineered for a local admin tool)
- In-memory (no persistence) — rejected (every page navigation would require re-login)

## Decision 6: Routing

**Decision**: Vue Router 4 with a navigation guard that checks `sessionStorage` before each route.

| Route      | Component          | Protected |
|------------|--------------------|-----------|
| `/login`   | `LoginView.vue`    | No        |
| `/`        | `ClientsView.vue`  | Yes       |
| `/:id`     | `ClientDetailView.vue` | Yes   |

Navigation guard behaviour:
- If navigating to a protected route with no session key → redirect to `/login`
- If navigating to `/login` with a session key → redirect to `/`

**Alternatives considered**:
- Single-view with conditional rendering — rejected (back-button behaviour becomes non-standard)

---

## Decision 7: Testing

**Decision**: Vitest 3 + `@vue/test-utils` 2, `jsdom` environment, coverage with `@vitest/coverage-v8`.

**Rationale**: Vitest is the standard test runner for Vite projects; it shares the same config
and transform pipeline, making test setup near-zero. `@vue/test-utils` is the official Vue
component testing library. `jsdom` simulates the DOM without a browser.

**Test scope**:
- Unit tests for `api.ts` (mock fetch, assert headers and error handling)
- Unit tests for `useSession.ts` composable (sessionStorage read/write/clear)
- Component tests for each view and reusable component (mount, assert rendering, simulate events)
- Navigation guard tests (routing logic)

**Alternatives considered**:
- Jest + `@vue/test-utils` — rejected (Vitest is the Vite-native choice; no separate jest transform config needed)
- Playwright (E2E) — deferred to a later feature; unit + component tests are sufficient for now

---

## Decision 8: Docker Serving Architecture

**Decision**: New `admin-ui` Docker service (multi-stage build: Node 22 Alpine → nginx Alpine)
served at port 80 inside the container. The existing nginx SSL proxy routes `/admin/` to this
service.

**Multi-stage Dockerfile stages**:
1. `node:22-alpine` — run `npm ci && npm run build` → produces `/app/dist`
2. `nginx:alpine` — copy `dist/` to `/usr/share/nginx/html`; serve with SPA fallback
   (`try_files $uri $uri/ /index.html`)

**nginx.conf changes** (existing `nginx/nginx.conf`):
```nginx
location /admin/ {
    proxy_pass http://admin-ui:80/;
    proxy_set_header Host $host;
}
```

The `docker-compose.yml` adds:
```yaml
admin-ui:
  build:
    context: ./admin-ui
    dockerfile: Dockerfile
  healthcheck:
    test: ["CMD", "wget", "-qO-", "http://localhost:80/healthz"]
    ...
```

**Alternatives considered**:
- Copy SPA `dist/` into existing nginx container at build time — rejected (couples two
  independent builds; SPA rebuild requires rebuilding nginx image)
- Serve SPA from a CDN / object storage — deferred (over-engineered for local dev/staging)
