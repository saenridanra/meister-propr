# Implementation Plan: React to Mentions in PR Comments

**Branch**: `009-react-to-mentions` | **Date**: 2026-03-20 | **Spec**: [spec.md](spec.md)  
**Input**: Feature specification from `/specs/009-react-to-mentions/spec.md`

---

## Summary

Adds a continuous background mention-scanning pipeline to MeisterProPR. Two new singleton `BackgroundService` workers — `MentionScanWorker` (producer) and `MentionReplyWorker` (consumer) — use a `Channel<MentionReplyJob>`-based queue backed by three new PostgreSQL tables. The scanner discovers recently-updated PRs via ADO's `minLastUpdateDate` filter (per-project watermark), detects `@<bot>` mentions using GUID-first / display-name-fallback logic, and enqueues reply jobs. The consumer fetches full PR context, calls a new `IMentionAnswerService` AI interface, and posts answers as inline thread replies using a new `IAdoThreadReplier` infrastructure capability. No new REST endpoints.

## Key Design Decisions

### Channel Registration (Program.cs)

```csharp
builder.Services.AddSingleton(
    Channel.CreateBounded<MentionReplyJob>(new BoundedChannelOptions(1000)
    {
        FullMode = BoundedChannelFullMode.Wait,
        SingleReader = true,
        SingleWriter = false,
    }));
```

Both workers receive the `Channel<MentionReplyJob>` singleton and access `.Reader` / `.Writer` directly.

### Startup Hydration (MentionReplyWorker.StartAsync)

Before `ExecuteAsync` starts:
1. Call `IMentionReplyJobRepository.ResetStuckProcessingAsync()` — transitions any `Processing` jobs back to `Pending` (handles crash-during-processing).
2. Load all `Pending` jobs from `IMentionReplyJobRepository.GetPendingAsync()`.
3. Write each to the channel.

### Scan Loop (MentionScanService.ScanAsync)

```
foreach crawl config (client-scoped):
  projectScan = GetProjectScanAsync(configId) ?? DateTimeOffset.UtcNow.AddHours(-1)
  recentPrs   = IActivePrFetcher (projectId, updatedAfter: projectScan.LastScannedAt)
  foreach pr in recentPrs:
    prScan = GetPrScanAsync(configId, repoId, prId)
    if prScan exists && pr.LastUpdatedAt <= prScan.LastCommentSeenAt → skip (log)
    threads = FetchExistingThreadsAsync(prId)
    latestTimestamp = DateTimeOffset.MinValue
    foreach thread / comment:
      latestTimestamp = max(latestTimestamp, comment.UpdatedAt)
      if MentionDetector.IsMentioned(comment.Content, reviewerGuid, displayName)
        && not ExistsForCommentAsync(clientId, prId, threadId, commentId):
          AddAsync(job); ChannelWriter.WriteAsync(job)
    UpsertPrScanAsync(configId, repoId, prId, latestTimestamp)
  UpsertProjectScanAsync(configId, now)
```

### AI Answer Prompt (AgentMentionAnswerService)

- **System**: "You are a PR review assistant. Answer the developer's question concisely and directly, grounded only in the PR content provided. Do not initiate a full review."
- **User**: PR title + description + up to 10 file diffs (truncated at 200 lines each) + existing thread summaries + `Question: {job.MentionText}`

### Deduplication

`ExistsForCommentAsync(clientId, prId, threadId, commentId)` is checked before enqueuing. The unique DB constraint on `mention_reply_jobs(client_id, pull_request_id, thread_id, comment_id)` provides the authoritative guard against duplicates.

## Complexity Tracking

| Item | Why Needed | Simpler Alternative Rejected Because |
|---|---|---|
| `Channel<MentionReplyJob>` alongside DB-poll in `ReviewJobWorker` | Immediate consumer wakeup; idiomatic .NET 10; spec-mandated | DB-poll at 2s adds unnecessary latency; spec explicitly chose Channel; `ReviewJobWorker` migration noted as future work |
| Two watermark tables + one job table | Separate concerns: scan state (long-lived) vs job state (short-lived, mutable) | Single monolithic table would conflate different lifecycles and make watermark updates racy with job state transitions |

## Migration Plan

One new EF Core migration: `AddMentionScanTables`

Creates:
- `mention_reply_jobs` — unique constraint on `(client_id, pull_request_id, thread_id, comment_id)`; FK to `clients`
- `mention_project_scans` — unique on `crawl_configuration_id`; FK to `crawl_configurations` (cascade delete)
- `mention_pr_scans` — unique on `(crawl_configuration_id, repository_id, pull_request_id)`; FK to `crawl_configurations` (cascade delete)

No changes to existing tables. `Down()` drops all three new tables.
